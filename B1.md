# B1: SWIM ↔ Raft Bridge Protocol

## 1. SWIM is the single source of truth for node liveness

**Why:** Raft and SWIM can diverge — Raft still counts a dead node as a voter while SWIM has declared it dead. Mixing
two sources of truth causes split-brain on membership decisions.

**Implementation:** Raft detects peer unresponsiveness faster (heartbeat ~150ms vs SWIM ~6s). Rather than act on it,
Raft signals SwimActor with the node ID to trigger an accelerated indirect probe. SWIM still makes the final Dead/Alive
call.

---

## 2. Do not reorganize Raft groups based on indirect reachability

**Why:** Raft's election mechanism already handles partial partitions correctly. Reorganizing groups based on a
reachability graph duplicates that logic at a higher layer with no benefit.

---

## 3. Add a bridge layer between Topology and Coordinator/Raft

**Why:** When a Raft group loses quorum, the Coordinator (whose state lives in that Raft log) is also frozen. The bridge
and Topology are the only layers still functional. A dedicated layer keeps concerns separated — Topology knows who is
alive, bridge knows how to act on Raft groups.

**Implementation:**

- Raft groups push quorum-loss notifications to the bridge (not polled)
- Bridge consults Topology for reachable replacement nodes
- Bridge issues ConfChange to form a new Raft group
- Scope: cluster-wide singleton (analogous to TiKV's Placement Driver)

---

## 4. Bootstrap safety rule for new Raft group after quorum loss

**Why:** A follower's log may be stale — it might have missed entries committed by a quorum that excluded it.
Bootstrapping from stale state causes permanent data loss.

**Decision:** Bootstrap only if the survivor was the **leader** of the old group, or a **quorum of the old group is
alive**.

- Leader is safe: Raft election protocol guarantees a leader's log contains all committed entries.
- Quorum alive: collectively guarantees all committed entries are reachable across survivors.
- Otherwise: wait until the condition is met. Do not accept data loss.

**Implementation:**
| Scenario | Safe to bootstrap? | Action |
|---|---|---|
| Leader alive, followers dead | ✓ | Bootstrap new group from leader's RocksDB state immediately |
| Follower alive, leader+others dead | ✗ | Wait for quorum to recover |

---

## 5. Stale file cleanup on recovered nodes

**Why:** Nodes that were dead and replaced by a new Raft group will have stale group files on disk when they recover.
They must not re-join the old group.

**Decision:** Lazy cleanup — on recovery, the node attempts contact and receives a group ID mismatch rejection, then
self-cleans. No explicit decommission message needed.
