# B2: WAL → Segment Transition: Atomic Completion Marker

## 1. Append a WAL fence record to data.seg on each batch flush

**Why:** "Fully reflected in data.seg" is not checkable without scanning the entire file. A crash between fsync and WAL
deletion leaves ambiguous state — re-applying the WAL duplicates records, skipping it loses records. The fence gives a
precise, checkable recovery point without a full scan.

The fence is the last record appended in each batch. Its presence guarantees all preceding records in that batch are on
disk.

---

## 2. Fence record contains the WAL offset it covers

**Why:** On recovery, the system must know which WAL entries have already been applied. The fence stores `wal_offset` so
recovery can re-apply only the unapplied tail of the WAL, not the entire log.

---

## 3. Recovery procedure

```
1. Scan data.seg backward to find the last valid fence
2. Truncate data.seg to that fence position  (discard partial writes after it)
3. Re-apply WAL entries from fence.wal_offset onward
4. Append new fence + fsync
5. Delete WAL entries up to new fence.wal_offset
```

**Why truncate instead of skip:** A crash mid-fsync can leave partial records after the last fence. Re-applying the WAL
onto partial data would produce corrupt or duplicate records. Truncating first makes the state clean before
re-application.

---

## 4. Variable-length records with trailing length field

Record format:

```
[crc: u32][type: u8][length: u32][payload: bytes][length: u32]
                                                  ↑ trailing length for backward scan
```

Backward scan: read trailing 4 bytes → jump back by `(length + header_size)` → repeat until fence found.

**Why variable-length over fixed-size blocks:**

- EastGuard is a message queue with widely varying record sizes. Fixed-size blocks cause high internal fragmentation — a
  small record wastes most of a block.
- Fixed-size blocks require chaining for large records (multi-block traversal), adding implementation complexity that
  variable-length avoids entirely.
- The only advantage of fixed-size blocks — natural O_DIRECT alignment — is trivially handled in variable-length by
  padding the batch once at flush time: `align_up(batch_size, BLOCK_SIZE)`. No per-record cost.
- Consumer reads use physical byte offsets from the RocksDB index and read forward — random access by block index is
  never needed, so fixed-size blocks give no benefit there either.

---

## 5. O_DIRECT alignment is handled at flush time, not per record

**Why:** O_DIRECT requires block-aligned writes. With variable-length records, the batch accumulator pads the entire
batch to the block boundary before flushing. This is a single `align_up` call per flush, not per record.
